# Code Review — 2026-05-28

Post-Codex-refactor review. Code is the source of truth. No code was changed.

---

## What is good

**Core module decomposition is clean.**
`run.py` is now genuinely thin — just CLI argument parsing and `main()`. Work is correctly distributed across `lifecycle.py` (run execution), `runtime.py` (assembly), `run_config.py` (config), and `workspace.py` (I/O). Each file has a single well-named responsibility.

**`RunConfig` and `RoleConfig` are typed Pydantic models.**
Config is no longer passed as opaque dicts through the lifecycle entry points. `RunConfig` is frozen, field-complete, and validated at construction time. `RoleConfig` is recursively typed (the `fallback` field is itself a `RoleConfig | None`), which correctly models arbitrarily deep fallback chains without any parallel structure.

**`PlayAttemptRecord` and `_PlayGameContext` are properly typed.**
Both are dataclasses with named fields. `PlayAttemptRecord` gives telemetry a stable typed surface. `_PlayGameContext` isolates per-game setup from the attempt loop — a correct factoring that prevents state from leaking between attempts.

**CreateGame Red challenge is a real improvement.**
The `CREATE_GAME_RED` loop in `engine.py` adds a self-critique step to `GameSpec` derivation without touching the adapter contract or the orchestration spine. The failure mode (unparseable Red output → accept as-is) is documented and reasonable.

**Blackboard audit trail is comprehensive.**
`games.jsonl` now captures `create_game`, `play_game`, and `integration` events in chronological order per run. All entries are sanitized before write. The file is append-only. This gives full visibility into what the pipeline did without exposing the blackboard to model prompts.

**`StateService.apply_update` removed.**
The API is now unambiguous: `apply_delta` is the only mutation path. There is no longer a proposal-workflow method competing with the runtime integration path.

**Boundaries are maintained.**
`run.py` contains no orchestration. `orchestration.py` contains no prompt rendering. `game/engine.py` contains no adapter-specific logic. The separation is real, not just nominal.

## What is bad

**`_PlayGameContext` uses `Any` for callback fields.**
`debug_event_fn`, `render_red_prompt_fn`, `render_referee_prompt_fn`, `verify_candidate_fn`, and `state_view` are all typed `Any` in the dataclass (`game/roles.py:54–69`). This defeats the type checker at the exact boundary between `engine.py` and `attempt.py`. Any mismatch in callback signature is invisible until runtime.

**Red finding schema is defined twice.**
`_CREATE_GAME_RED_FINDING_SCHEMA` in `engine.py` and `_RED_FINDING_SCHEMA` in `roles.py` are structurally identical dicts. They will silently diverge under maintenance. One definition should live in `roles.py` and be imported by `engine.py`.

**`engine.py` re-exports private telemetry symbols.**
`engine.py:__all__` exports `_append_integration_to_blackboard` and `_append_northstar_proposal_to_blackboard` — both underscored private names — so `orchestration.py` can import them through `engine`. The privacy convention is broken: either the names should be public (remove the underscore) or `orchestration.py` should import directly from `telemetry.py`.

**`generate_with_tools` failure is silently swallowed.**
In `attempt.py:86–89`, a `ValueError` from `blue_role.generate_with_tools(...)` causes silent fallback to text generation with a bare `except ValueError: pass`. If the failure is a schema mismatch or a model misconfiguration, the caller gets text output instead of an error, making failures hard to diagnose.

**`GitRepositoryArtifactAdapter` is a dead code path.**
`state.py` registers a `GitRepositoryArtifactAdapter` in `build_default_state_artifact_registry()`, but `resolve_project_type_adapter` raises `ValueError` for `project_type='git'`. The adapter kind is present in the registry but the project type cannot be instantiated. This creates confusion about whether `git` is supported.

**`CodingArtifact.language` defaults to `"python"` in the model.**
`state.py:148` has `language: str = "python"`. The adapter enforces the field as required at creation time, but the model default means loading old state from JSON without a `language` field silently substitutes `"python"` rather than surfacing a validation error. The default exists to tolerate legacy state but masks schema drift.

## What is still in flux

**`_build_play_game_context()` callback injection pattern.**
`engine.py` passes `debug_event`, `_render_red_prompt`, `_render_referee_prompt`, and `_verify_candidate_with_adapter` as untyped callback arguments into `_PlayGameContext`. This pattern suggests the boundary between `engine.py` and `attempt.py` hasn't fully settled. It may be moving toward formal injection (Protocol types, dependency inversion) or consolidation back into a single module.

**`max_create_game_attempts` is hardcoded.**
`create_game()` accepts `max_create_game_attempts: int = 2` as a parameter defaulting to 2. Unlike `max_sub_gaps` and `max_depth`, which are config-driven via `RunConfig`, this knob isn't exposed to the spec. Whether it belongs in config alongside the other bounds hasn't been decided.

**CREATE_GAME_RED role integration.**
The role is functional and tested, but its schema lives in `engine.py` while the PlayGame Red schema lives in `roles.py`. The responsibility split between engine and roles for schema ownership is inconsistent and may consolidate in one direction.

## Recommended next steps

Priority order:

1. **Type the callback fields in `_PlayGameContext`.**
   Define `Protocol` types (or `Callable[...]` signatures) for `debug_event_fn`, `render_red_prompt_fn`, `render_referee_prompt_fn`, and `verify_candidate_fn`. Replace `Any` with the typed signatures. Fix `state_view: Any` → `state_view: StateView`. This is the highest-leverage typing gap: it covers the attempt loop boundary, which is the most exercised code path.

2. **Deduplicate Red finding schemas.**
   Move `_CREATE_GAME_RED_FINDING_SCHEMA` out of `engine.py` and import `_RED_FINDING_SCHEMA` from `roles.py` in its place. One definition, one location.

3. **Move `max_create_game_attempts` into `RunConfig`.**
   Add a `max_create_game_attempts: int = 2` field to `RunConfig` alongside `max_sub_gaps` and `max_depth`. Pass it through from `_solve_gap` into `create_game`. This makes all loop bounds configurable from the spec.

4. **Fix the `engine.py` re-export of private telemetry symbols.**
   Remove `_append_integration_to_blackboard` and `_append_northstar_proposal_to_blackboard` from `engine.py:__all__`. Update `orchestration.py` to import them directly from `baps.game.telemetry`. Either keep the underscore and treat them as genuinely private, or remove it and treat them as package-internal public API.

5. **Remove `GitRepositoryArtifactAdapter` or gate it properly.**
   The adapter is registered but the project type is not implemented. Either remove the adapter from `build_default_state_artifact_registry()` until the project type is live, or implement the project type. The current state is misleading to anyone reading `state.py`.
